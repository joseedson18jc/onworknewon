## Prioritized findings

### 🚨 P-CRITICAL — Security (fix today)

| # | Issue | Severity | Evidence |
|---|-------|----------|----------|
| 0 | **OpenAI API key (`sk-proj-…`, 164 chars) hard-coded in the client-side inline JS** of the production page. GitHub's secret scanner blocked committing the DOM snapshot as evidence. The page also ships what look like **ElevenLabs and Surfe API keys** (multiple 60–200 char opaque tokens inside quoted string literals in the inline bundle). Any visitor to the site can extract these from "View Source" and run up unlimited charges on your OpenAI / ElevenLabs / Surfe accounts. | 🚨 **Critical** | `dom.html:6`, `dom.html:5787`, `dom.html:5930` (not committed — see `reports/screenshots-2026-04-16/` locally; file is gitignored) |

**Immediate remediation (within hours, not days):**

1. **Rotate the OpenAI key, ElevenLabs key, and Surfe key right now.** Revoke the old keys from each provider's dashboard.
2. Put calls to OpenAI / ElevenLabs / Surfe behind a **server-side proxy** (a Vercel serverless function in `/api/*`). The browser sends *user* credentials (session cookie / JWT), the server attaches the provider API key from `process.env`. Never expose provider keys to the client.
3. Add a rate-limit + per-user quota on the proxy so a compromised session can't drain the account.
4. Add **CI secret scanning** (e.g. Gitleaks, `trufflehog`, or GitHub's own advanced security) on every push.
5. Set usage/billing alerts on each provider (OpenAI: hard monthly cap; ElevenLabs: per-minute budget; Surfe: request cap).

This is the single most important finding in this audit. Everything else below is a normal UX/perf/a11y lift; this one is an active financial/exfiltration risk.

### P0 — Accessibility & correctness (ship blockers)

| # | Issue | Severity | Location | Suggested fix |
|---|-------|----------|----------|---------------|
| 1 | **Bilingual text concatenated in a single `<h1>`** — both languages live in the DOM at the same time (`<span class="lang-pt">…</span><span class="lang-en">…</span>`). Screen readers read both. | 🔴 High (a11y + SEO) | every heading, copy block, CTA | Add `aria-hidden="true"` to the hidden-language span, and sync `<html lang>` when the user toggles. Better: render only the active language via JS + `data-i18n` keys. |
| 2 | **30 `<input>` elements without `<label>`** — including `forgotEmail`, `briefSearch`, `spJob`. Only `placeholder=""` is used. | 🔴 High | sign-in, forgot-password, 🔒 search bars, Surfe filters | Add `<label for>` or `aria-label`. Placeholders disappear on focus and fail WCAG 3.3.2. |
| 3 | **8 `<button>` elements with no visible text and no `aria-label`** (likely icon-only buttons). | 🔴 High | 🔒 sidebar, chart actions | Add `aria-label="…"` per button. |
| 4 | **Two `<h1>` elements on the same page** ("Bem-vindo de volta" and "Inteligência para cada conversa comercial."). | 🔴 High | login + hero | Keep only one `<h1>`; demote the other to `<h2>`. |
| 5 | **Sign-in is not wrapped in a `<form>`** — no native Enter-to-submit, no password-manager autofill, no browser validation. | 🔴 High | login card | Wrap inputs in `<form onSubmit>` with `method="post"`, proper `autocomplete="username|current-password"`, `required`, and a `type="submit"` button. |
| 6 | **Only 2 landmarks** (`<aside>` + `<main>`). No `<header>`, `<nav>`, `<footer>`, no skip-link. | 🟠 Medium | layout shell | Add semantic landmarks; insert a "Skip to content" link as the first focusable element. |

## Headline takeaways

1. **Performance is green on first paint** — the single-file architecture works *because* the page is small. It will not stay green once authenticated views mount the full DOM tree and charts load.
2. **Accessibility is the biggest single lift.** The duplicated-in-DOM bilingual pattern cascades into dozens of WCAG issues at once — fix that one pattern and findings #1, #2 (partially), #4, #11 resolve together.
3. **Visual language is coherent but thin** — teal + navy gradient, one typeface, decent dark mode. Needs a full design system (neutral ramp, semantic colors, chart palette, typography scale) before the dashboard can feel professional on par with Ramp / Linear / Attio.
4. **The sign-in screen is too empty at laptop+** — 60%+ of the viewport is dead space. Phase 2 should propose a value-prop right pane with a chart-rich preview of what the user is logging into.
5. **Charts, illustrations, and motion** — the house style (Recharts/Nivo + Framer Motion + Lottie + unDraw) is a perfect fit once we're inside. Phase 2 will produce the token set + component inventory + chart menu tailored to sales intelligence dashboards.
